LLM Fundamentals

Welcome!

  • This workshop: AI Programming & Large Language Models
  • You have likely already used LLMs:
    • ChatGPT, Claude, Gemini, and others
  • By the end of this workshop, you will understand:
    • How LLMs work at a conceptual level
    • The difference between models, apps, and harnesses
    • Which AI coding tools to use and how to get started

What is an LLM?

  • Think of them as advanced pattern recognition systems
  • They have “read” massive amounts of text from the internet, books, and code
  • Based on those patterns, they can generate new text
  • LLMs really do one thing: predict the next token
  • Like an incredibly advanced version of autocomplete on your phone
  • But they cannot truly “think” or “understand” like humans

Tip

Despite their limitations, LLMs are genuinely useful for coding, writing, research, and analysis.

The Transformer Architecture

  • Modern LLMs are powered by the Transformer architecture
  • Introduced in “Attention is All You Need” (Vaswani et al., 2017)

How it works (simplified):

  1. Tokenization: Text is split into smaller units called tokens
  2. Relationship mapping: The model builds a mathematical representation of how tokens relate
  3. Attention mechanism: The model focuses on the most relevant parts of the input to generate each output token

Note

“The cat sat on the mat because it was warm.” – Attention helps the model understand that “it” refers to “the mat,” not “the cat.”

Tokens: The Building Blocks

  • LLMs process text as tokens, not whole words
  • A token can be:
    • A whole word (e.g., “cat”)
    • A subword (e.g., “un-”, “break”, “-able”)
    • An individual character or punctuation mark

Why tokens?

  • Handles unknown words by breaking them into known subwords
  • More efficient than processing character by character
  • Trade-off: tokenization works better for some languages than others

Tip

You can explore how text is tokenized at platform.openai.com/tokenizer.

Context Window: The LLM’s Memory

  • The context window is the number of tokens an LLM can process at once
  • Think of it as the model’s short-term memory

Larger context window:

  • “Remembers” more of the conversation
  • More coherent responses
  • More computational cost

Smaller context window:

  • Faster processing
  • May lose track of earlier parts
  • Less expensive to run

Warning

Managing context windows matters when working with AI tools! Break down problems into smaller chunks.

Training Phases

  • Training an LLM happens in three phases:

1. Self-Supervised Learning (Pre-training)

  • Trained on massive text datasets without explicit labels
  • Learns to predict the next token – learns grammar, facts, and patterns

2. Supervised Fine-Tuning

  • Trained on smaller, curated datasets for specific tasks
  • Makes the model better at following instructions

3. Reinforcement Learning from Human Feedback (RLHF)

  • Human evaluators rank model outputs by quality
  • Model learns to produce responses that are more helpful, honest, and harmless

LLM Architectures (Overview)

Architecture How it works Best for
Encoder-Decoder Understands input, then generates output Translation, summarization
Encoder-Only Focuses on understanding input Classification, search (BERT)
Decoder-Only Generates text token by token Chatbots, code generation (GPT, Claude)
Mixture of Experts Routes input to specialized sub-models Efficient scaling (fewer params active)

Tip

Most LLMs you interact with today (ChatGPT, Claude, Gemini) are decoder-only models, sometimes enhanced with Mixture of Experts.

Limitations

  • Hallucinations: LLMs can confidently generate incorrect information
  • Bias: Training data contains biases that the model can reproduce
  • No true understanding: Pattern matching, not reasoning from first principles
  • Knowledge cutoff: Models only know what they were trained on
  • Context limits: Can lose coherence in very long conversations
  • Sensitivity to prompts: Small changes in wording can produce very different results

Warning

Always verify important information from LLMs. They are tools, not oracles.

Models, Apps & Harnesses

A Framework for Understanding AI Tools

  • Based on Ethan Mollick’s framework from “One Useful Thing”
  • Three layers to understand:
  1. Models = the underlying AI “brains”
  2. Apps = the interfaces you interact with
  3. Harnesses = systems that give AI access to tools for autonomous work

Important

The same model can produce very different results depending on the harness it is placed in!

Models: The AI Brains

  • The foundation model is the core intelligence
  • Different providers train different models:
    • OpenAI: GPT-4o, o3, o4-mini
    • Anthropic: Claude Opus, Claude Sonnet
    • Google: Gemini Pro, Gemini Flash
    • Mistral: Mistral Large, Mistral Medium
    • Meta: Llama (open-source)

Warning

Free/default tiers often use weaker models. Always select the frontier or advanced model when available – the difference is noticeable.

Apps: The Interfaces

  • Apps are how you access the models
App URL Provider
ChatGPT chatgpt.com OpenAI
Claude claude.ai Anthropic
Gemini gemini.google.com Google
Le Chat chat.mistral.ai Mistral
  • All four offer free access to capable models
  • Paid tiers unlock the most powerful models and higher usage limits
  • Mistral offers an affordable student tier with access to Mistral Vibe (not flawless, but acceptable)
  • The app is just the interface – the model behind it matters most

Harnesses: The real differentiator

  • A harness is a system that wraps a model with tools and capabilities
  • Same model, different harness = completely different results

Examples of harnesses:

  • A chat window (Claude.ai) – conversational Q&A
  • A coding agent (Claude Code) – reads files, writes code, runs tests
  • An IDE integration (Copilot in Zed) – inline code suggestions

Tip

Pick the right harness for the task! A chat window is great for brainstorming, but a coding agent is better for building features.

AI providers today

Provider Model App Strength

Anthropic Claude Opus/Sonnet Claude.ai Reasoning, coding, long context

  • This changes fast – new models appear every few months
  • What matters: understand the model/app/harness distinction so you can evaluate new tools as they emerge

Why This Framework Matters

  • Question: Why does the same AI feel “smarter” in some tools than others?
  • Because the harness determines:
    • What context the model receives
    • What tools it can use (web search, file access, code execution)
    • How many steps it can take autonomously
    • How results are presented back to you

Note

When someone says “AI can’t do X,” ask: which model, in which harness? The answer often changes with the right setup.

AI Coding Tools

Homebrew: Your Package Manager

  • Homebrew is a package manager for macOS
  • Installs command-line tools and applications with a single command
  • No more hunting for download pages or dragging .dmg files

How to install Homebrew:

/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"

Then installing tools is as simple as:

brew install <tool-name>        # CLI tools
brew install --cask <app-name>  # GUI applications

Tip

We will use Homebrew to install all coding tools in this workshop. If you are not on macOS, download links are provided as alternatives.

Zed: A Modern Code Editor

  • Recommended IDE for this course
  • Modern, fast code editor built in Rust
  • Built-in AI assistant with multiple model support

Key features:

  • Supports cloud models (Claude, GPT) and local models (Ollama)
  • Great for pair programming with AI
  • Free and open-source
  • Fast, lightweight, and actively developed

How to install (macOS):

brew install --cask zed

For Linux or Windows, download from zed.dev.

Tip

Zed offers a free student tier with solid code completion and $10 in API credits for at least one year. Sign up at zed.dev with your university email.

GitHub Copilot

  • AI pair programmer integrated into editors (VS Code, Zed, JetBrains)
  • Free for students via GitHub Student Developer Pack
  • Provides inline code completion as you type

How to get access:

  1. Go to education.github.com/pack
  2. Sign up with your university email
  3. Verify your student status
  4. Wait for approval (usually 1-2 days)

Note

Copilot’s code completion can also be activated in Zed if you prefer it over Zed’s built-in completion.

OpenCode: Terminal AI Assistant

  • Open-source terminal-based AI coding assistant
  • Great for quick tasks directly in your terminal

How to install (macOS):

brew install opencode-ai/tap/opencode

For other platforms, see opencode.ai.

Key features:

  • Supports multiple AI providers (bring your own API key)
  • Lightweight and fast
  • Read, edit, and create files from the terminal
  • Good for quick fixes and file operations

Claude Code: Autonomous Coding Agent

  • Anthropic’s CLI coding agent
  • Goes beyond suggestions: autonomous multi-step task execution

How to install (macOS):

brew install claude-code

For other platforms, see claude.ai/code.

What makes it different:

  • Reads your entire codebase for context
  • Can plan, write, test, and debug code autonomously
  • Executes terminal commands and iterates on errors
  • Best for complex, multi-step tasks

Note

Claude Code requires an Anthropic API key and incurs usage costs.

Comparing AI Coding Tools

Tip

Start with Zed + Copilot for daily coding. Use Claude Code or OpenCode when you need to tackle larger tasks from the terminal.

Choosing the Right Tool

  • Question: When would you use a chat interface vs. a coding agent?

Chat interface (Claude.ai, ChatGPT)

  • Brainstorming ideas
  • Explaining concepts
  • Drafting text or outlines
  • Quick one-off questions

Coding agent (Claude Code, OpenCode)

  • Building features across files
  • Debugging complex issues
  • Refactoring codebases
  • Running and testing code

Important

Remember: same model, different harness = different capabilities. Pick the right tool for the job.

Tips & Wrap-up

Tips for Effective AI Pair Programming

1. Be precise in your instructions

  • The more specific your prompt, the better the result
  • Include context: language, framework, expected behavior

2. Always review generated code

  • LLMs make mistakes, suggest inefficient solutions, or miss edge cases
  • Your code, your responsibility

3. Iterate and refine

  • Use the output as a starting point
  • Ask follow-up questions to improve the result
  • Break complex requests into smaller steps

Manage Context Effectively

  • LLMs have a finite context window – treat it as a precious resource
  • Break down problems into manageable chunks rather than dumping everything at once
  • Be specific about which files or functions you need help with
  • Start fresh conversations when switching topics to avoid confusion
  • Provide relevant context upfront rather than expecting the model to guess

Tip

Think of context like a desk: keep only what you need for the current task on it, and clear it when you move to the next task.

Takeaways

  1. AI is a tool, not a replacement – you still need to understand the fundamentals
  2. Understand before accepting – always read and verify AI-generated output
  3. Pick the right harness – chat for brainstorming, agents for building, IDE for daily coding
  4. Models matter – always use the most capable model available to you
  5. Context matters – clear prompts and managed context give you better output

Warning

If you use free models, be aware that your prompts may be used by providers and are not private. For learning and experimenting, this should be no issue.
